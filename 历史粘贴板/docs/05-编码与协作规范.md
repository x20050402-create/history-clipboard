# 05 · 编码与协作规范

## 1. 代码风格

- 语言：JavaScript（Node + Electron），暂不引入 TypeScript / 构建工具，降低复杂度
- 注释：关键模块和函数用**中文注释**说明意图，贴合现有代码风格
- 命名：变量/函数 camelCase；文件名 kebab-case（如 `clipboard-watcher.js`）
- 模块化：一个文件一类职责（窗口、数据库、监听分开）
- 函数尽量短小、单一职责；避免一次性写超长文件

## 2. 项目结构约定

```
src/main.js              主进程入口
src/preload.js           安全桥接（白名单 API）
src/db.js                数据库
src/clipboard-watcher.js 剪贴板监听
src/renderer/            界面（html/css/js）
build/                   图标等打包资源
docs/                    标准文档
开发日志/                每日开发日志
```

## 3. 运行方式（重要）

本机 Node 装在 `C:\Program Files\nodejs`，Git Bash 默认不在 PATH。运行前需先加 PATH：

```bash
export PATH="$PATH:/c/Program Files/nodejs"
```

常用命令：

| 操作 | 命令 |
| --- | --- |
| 安装依赖 | `npm install` |
| 运行体验 | `npm start` |
| 打包 exe | `npm run dist` |

> 在 VS Code 里：打开内置终端 → 运行上面命令即可。

## 4. 安全约定

- 渲染进程开启 `contextIsolation`、关闭 `nodeIntegration`
- 界面只通过 preload 暴露的白名单 API 与主进程通信
- 不记录敏感剪贴板内容；数据只存本机，不联网上传

## 5. Git 提交约定（如启用版本管理）

- 提交信息格式：`类型: 简述`，类型用 `feat / fix / docs / refactor / chore`
- 示例：`feat: 实现文字剪贴板监听`、`docs: 补充设计规范色板`
- 小步提交：每完成一个开发小步提交一次

## 6. 工作闭环（每次开发遵循）

1. **开发前**：读 `docs/04-开发步骤.md` 找到"当前进度 / 下一步"
2. **改 UI 前**：先看 `docs/03-设计规范.md`
3. **开发中**：小步实现 + 本地运行验收
4. **开发后**：更新当天 `开发日志/YYYY-MM-DD.md`（完成 / 待办 / 备注），并在 `04-开发步骤.md` 勾选进度
5. 软件可运行体验时，明确告知用户运行方式
